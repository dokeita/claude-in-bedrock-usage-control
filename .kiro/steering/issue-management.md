# Issue 管理ルール

## 概要

Issue は `issues/` ディレクトリ配下に Markdown ファイルとして管理する。

## ディレクトリ構成

```
issues/
├── pending/    # 未着手・対応中の Issue
└── closed/     # 完了した Issue
```

## ファイル命名規則

- 形式: `NNNN-簡潔なタイトル.md` (例: `0001-usage-limit-notification.md`)
- 連番4桁 + ハイフン + ケバブケースのタイトル

## Issue ファイル作成ルール

Issue ファイルは必ず 5W1H を意識して以下のテンプレートに従って作成すること。

```markdown
# タイトル

## Why（なぜ）
この Issue が必要な理由・背景・課題

## What（何を）
達成すべきこと・ゴールの定義

## Who（誰が）
担当者・関係者

## When（いつ）
期限・優先度

## Where（どこで）
対象のコンポーネント・ファイル・サービス

## How（どうやって）
実装方針・アプローチ

## 完了条件
- [ ] 具体的な完了条件をチェックリストで記載
```

## 運用ルール

- 新規 Issue は `issues/pending/` に作成する
- 完了した Issue は `issues/closed/` に移動する
- 1 Issue = 1 ファイル。複数の課題を1ファイルにまとめない
- テンプレートの各セクション（5W1H）は省略せず必ず記載する
